## 面试问答

下面根据原始笔记和标准技术答案，把每个问题补充为更规范、可答复的版本，便于面试时直接使用。

- **遇到钓鱼邮件时的处理方法是什么？**
	- 保留原始邮件（不要删除），导出完整邮件头（raw/原始消息）。
	- 立即隔离受影响主机（如果是内部账号被点开），更改相关账号密码并开启多因子认证。
	- 根据邮件头中的各层 `Received` 字段追踪来源 IP、检查 SPF/DKIM/DMARC 验证结果；将可疑 IP/域提交到威胁情报平台并比对已知 IOC。
	- 向安全运营/管理层上报，并将邮件样本上传到沙箱进行静态/动态分析（在隔离环境中）。

- **如何溯源邮件真实来源 IP？**
	- 从邮件原始头部（Headers）提取最早的 `Received` 条目，通常靠近底部的那一行最接近发送者真实网络。注意伪造和中继服务器。 
	- 检查邮件发送链路中的邮件服务器日志（如 Postfix/Exim/Exchange 的 mail.log 或 Message Tracking）。
	- 使用被动 DNS、WHOIS、SSL/证书透明度、威胁情报平台和黑名单服务交叉验证域名与 IP 的历史关系。

- **如果攻击者使用 CDN/域前置，如何定位真实源？**
	- 对邮件来说，优先查看邮件头和中继日志；对于网站被域名前置，常用的溯源手段包括被动 DNS（历史解析）、证书透明度记录、子域/子主机暴露、以及第三方情报（Shodan、Censys、PassiveTotal）。
	- 检查是否有未通过 CDN 的接口（如 API、内网回连、老域名、历史解析记录），合作 ISP 或托管提供商可协助查询源 IP。
	- 注意合法合规：优先通过日志与情报渠道核实，不要进行未授权的主动攻击性扫描。

- **基线排查（攻击场景）一般包括哪些工作？**
	- 初始响应：确认告警范围、隔离受影响系统（按业务优先级）、保全证据（内存快照、网络流量、系统日志）。
	- 收集痕迹：系统事件日志、应用/中间件日志、网络连接（netstat/ss）、计划任务、自启动项、可疑进程、已安装服务和新建用户记录。
	- 恶意软件分析：提取可疑文件，放入隔离环境进行静态/动态分析；使用防病毒/EDR 快速查杀与溯源。
	- 横向排查：基于日志和网络拓扑查找可能被横向渗透的主机，查找同一时间窗内的异常行为。
	- 恢复与根因：补丁/加固、改密、恢复业务、出具事件报告并总结改进措施。

- **Webshell 无法出网（分析受限）怎么办？**
	- 优先在受控的隔离环境内收集样本并做静态分析（hash、字符串、文件行为）；对动态分析使用内网隔离的沙箱或跳板主机进行受控执行。
	- 可通过企业内部跳板/堡垒机、安全分析平台导出流量或在受控环境下模拟出网行为；若必须使用隧道类工具，先评估合规和风险，并在授权和隔离条件下进行。

- **如何辨别常见 Webshell 工具的流量特征？**
	- 观察异常 HTTP 请求模式：非标准或固定的 URI、长请求体、频繁的 POST 到特定接口、请求携带加密或 base64 编码的 payload。
	- 非正常 User-Agent、Referer 或自定义 Header；不符合常规业务的周期性 beaconing（心跳）；异常的响应长度或响应中包含 shell 交互标志。
	- 结合已知 IOC（URI 模式、特征字符串、文件名）和 NIDS/IDS 策略进行匹配与检测。

- **如何在流量中判断 SQL 注入？**
	- URL 或参数中出现 SQL 关键字片段（`' OR 1=1`、`UNION SELECT`、`--` 等）、不合常理的响应错误（数据库错误栈）、时间盲注的响应时延差异。
	- 观察异常的数据库查询模式、重复请求试探、以及与应用日志的异常错误条目关联。

- **CSRF 与 SSRF 的主要区别？**
	- CSRF（跨站请求伪造）：利用受害者浏览器的认证上下文，诱导用户在已认证的站点上执行非本意操作。防护：使用 CSRF token、严格的 SameSite Cookie、验证来源/Referer。
	- SSRF（服务端请求伪造）：攻击者让服务器去请求任意资源（内部或外部），可能访问内网服务或本地元数据服务。防护：白名单目标、出站请求代理、最小化服务权限、网络分段。

- **如何判断目标主机是 Windows 还是 Linux？**
	- 通过开放端口与响应（如 SMB/NetBIOS/445 更可能是 Windows）、服务名、返回的文件路径（C:\ 与 /var/log）、系统日志风格、以及远程命令返回的命令行工具名判断。

- **Windows 上的重点高危端口有哪些？**
	- 常见关注端口：135（RPC）、139/445（SMB）、3389（RDP）、5985/5986（WinRM）、5988/5989（WBEM）、1433（SQL Server）。

- **对 Spring Boot 的理解要点？**
	- Spring Boot 是基于 Spring 的快速开发框架，特点包括自动配置（Auto-configuration）、starter 依赖、内嵌容器（Tomcat/Jetty）、Actuator 监控端点、基于配置文件的参数化。
	- 安全关注点：暴露的 Actuator 端点、默认配置泄露、依赖库反序列化风险、敏感配置（如管理端口、凭证）需加固与限制访问。

- **Linux 常见日志存放位置？**
	- 系统通用：`/var/log/`（如 syslog、messages、auth.log、kern.log）；systemd 系统可用 `journalctl` 查看二进制日志；中间件日志通常在 `/var/log/<service>/` 或服务配置指定路径。

- **常见中间件举例？**
	- Web/应用：Nginx、Apache(httpd)、IIS、Tomcat
	- 数据库/缓存：MySQL、PostgreSQL、Redis、MongoDB
	- 其他：Load Balancer、消息队列（RabbitMQ、Kafka）、反向代理/缓存（Varnish）

- **排查中间件日志时注意点？**
	- 关注 4xx/5xx 错误、异常堆栈、上传点、非预期请求路径和带有 SQL/命令片段的请求参数；使用 grep/awk 等工具定位可疑请求并结合时间窗口做溯源。

- **关于 Shiro (rememberMe) 漏洞的说明（如低版本）**
	- 早期某些 Apache Shiro 配置使用静态或可预测的 `rememberMe` 加密密钥，攻击者可以利用已知密钥构造反序列化 payload 导致远程代码执行。修复手段包括升级到安全版本、确保随机且安全的密钥、禁用不必要的反序列化行为、限制外部输入的反序列化。

- **最深的漏洞经验（面试答法提示）**
	- 用一两个你实际参与调查或修复的真实案例来描述：场景、证据、根因、处置和最终改进点。突出你的取证与复盘能力，而不是只谈漏洞名字。

- **SRC/代码挖掘经验回答要点**
	- 说明你用过的工具（如 grep、ripgrep、Semgrep、搜索敏感字符串/入口点）、常用策略（识别反序列化入口、输入校验缺失、参数解析路径）和一个简短示例。

- **Linux 常用基础命令（面试可背诵）**
	- `ps`, `top`, `ss`/`netstat`, `lsof`, `find`, `grep`, `awk`, `sed`, `last`/`lastb`, `journalctl`, `tar`, `scp`, `ssh`, `chmod`, `chown`, `iptables`/`nft` 等。

- **XXE（外部实体注入）简述**
	- XXE 利用不安全的 XML 解析器，通过外部实体引用读取本地文件或向外部发送请求。防护：禁用外部实体解析、使用安全解析器或对输入进行严格校验。

- **SSRF 的防御手段举例**
	- 使用目标白名单、限制出站请求的网络策略（网络分段/ACL）、通过跳板/代理进行强制审计、对 URL 参数做严格校验并禁止访问内网敏感地址（如 169.254.169.254）。

- **Cookie 与 Session 的区别**
	- Cookie 存储在客户端（浏览器），可包含会话 ID 或持久化数据；Session 通常存储在服务器端，Cookie 用于保存 Session ID 以便服务器识别会话。

- **传输层的协议有哪些？**
	- 常见传输层协议：TCP、UDP（以及较少见的 SCTP）。TLS/SSL 是基于传输层的安全层/加密协议，运行在 TCP 之上。

- **学习经历（回答面试时的建议结构）**
	- 简短列出你的主要学习路径、重要实战/项目、常用工具和你持续学习的方式（阅读白皮书、CTF、实战演练、订阅威胁情报）。

- **Fastjson 风险与防护**
	- Fastjson 曾发生多次因自动反序列化导致的 RCE 漏洞。防护：升级到安全版本、关闭 autoType、使用白名单、限制反序列化入口、必要时替换为更安全的序列化库。

- **关于‘银狐木马’的面试回答要点**
	- 真实回答应聚焦检测与响应：列出典型 IOC（文件名、注册表项、网络行为）、检测方法（EDR、流量分析）、应对流程（隔离、取证、清除、恢复）。不要声称会主动使用恶意工具。

- **正向代理与反向代理的区别（简洁回答）**
	- 正向代理：客户端配置代理，由客户端主动通过代理访问外部资源（代表客户端）。
	- 反向代理：代理位于服务器侧，代表服务器接收外部请求并转发到内部服务器（隐藏真实后台服务器）。
